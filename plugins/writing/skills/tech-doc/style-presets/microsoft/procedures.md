# Microsoft — Procedures and instructions

Source: https://learn.microsoft.com/en-us/style-guide/procedures-instructions/
Last refreshed: 2026-04-29

## Prerequisites

The best procedure is one the reader does not need. If the UI is self-explanatory, skip the procedure. When a procedure is necessary, make sure the reader has everything required before they begin: software, permissions, accounts, or configuration. State prerequisites before step 1, not inside steps.

## Step structure

- One instruction per step. Combine two actions only when they occur in the same UI location and the combination does not cause confusion.
- Make sure the reader knows where to act before you describe the action: "For **Alignment**, select **Left**." Not "Select **Left** for **Alignment**."
- Use a complete sentence for each step. Capitalize the first word; end with a period.
- Start with an imperative verb most of the time. When context is needed first, lead with a brief orienting phrase: "On the ribbon, go to the **Design** tab."
- State the result after the action when the outcome may not be obvious.
- Conditions precede the instruction: "If the dialog is open, select **Cancel**." Not "Select **Cancel** if the dialog is open."
- Optional steps are labeled "Optional:" at the start.

## Numbering and bullets

- Use a numbered list for multi-step procedures.
- For a single-step procedure, a bullet is acceptable when you want visual consistency with numbered procedures elsewhere in the same document.
- A single-step procedure can also be a plain sentence or short paragraph when list formatting would feel heavy.
- Abbreviate simple sequential selections with right angle brackets: "Select **Accounts** > **Other accounts** > **Add an account**." Include a space before and after each bracket; do not bold the brackets. Check accessibility impact before using this pattern.

## Procedure introductions

- Introduce a procedure with a sentence that adds information beyond the heading. Do not restate the heading word for word.
- Use headings to help readers find procedures. Keep headings parallel in structure and concise.
- If a procedure has a goal statement, lead with "To \<do X\>:" and follow immediately with the numbered steps.
- If intervening content (a note, a paragraph) separates the introduction from the steps, end the introduction with a period, not a colon.

## Tone in procedures

- Active voice, imperative mood: "Select", "Enter", "Open". Not "You will need to select" or "The user should open".
- Second person ("you") when a subject is needed.
- Present tense throughout.
- No "please".
- Prefer input-neutral verbs ("select", "choose", "go to") over device-specific verbs ("click", "tap") so instructions work for mouse, keyboard, touch, and voice.

## Single-step vs multi-step

- Multi-step (two or more sequential actions): numbered list.
- Single-step: a bullet for visual consistency, or a plain sentence when list formatting is unnecessary overhead.
- Aim to keep all steps visible on one screen without scrolling. If a procedure grows beyond roughly seven steps, look for sub-procedures or split the task.

## Parenthetical UI hints

When a UI element label is ambiguous or appears in more than one place, add a short parenthetical after the label: "Select **Design** (the tab, not the menu)." Keep it to one brief clause. If more explanation is needed, add a screenshot or restructure the content.

## Sub-steps

When a step has sub-actions that belong together logically but are distinct enough to list, use a lettered sub-list (a, b, c) beneath the parent step. Keep sub-steps to a minimum. More than two levels of nesting (step > sub-step) is a signal the procedure needs restructuring.
